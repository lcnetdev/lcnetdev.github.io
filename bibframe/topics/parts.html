Parts
BIBFRAME Topics and Issues

Back to Topics and Issues Home

A BIBFRAME Work or Instance might have various parts, for example an abstract or libretto. While each of these various parts is a component of the Work or Instance, it may be distinguishable as a resource itself, and as such we may be able to make statements specifically about that part.

For example a book written in English might have an abstract written in French. BIBFRAME currently handles this by allowing the property bf:language to indicate what part of the Work the language applies to: property bf:languageOfPart is a property of bf:Language.  (Its value is a string, for example ‘abstract'.) It is suggested that this be treated instead by using properties hasPart and isPartOf, as illustrated in the following example.

Example 1.

<http://bibframe.example.con/works/books/goldfinch> a bf:Work ;
      bf:title "The Goldfinch" ;
      bf:language  <http://id.loc.gov/vocabulary/iso639-1/en>
      bf:hasPart <http://bibframe.example.con/workParts/abstracts/goldfinch> .

<http://bibframe.example.con/workParts/abstracts/goldfinch> a bf:Work ;
      bf:title "The Goldfinch - Abstract" ;
      bf:isPartOf  < http://bibframe.example.con/works/books/goldfinch >.
      bf:language  <http://id.loc.gov/vocabulary/iso639-1/fr> .

Other examples where a statement applies to a specific part of a resource:

  • A resource might have a section of photographs. The intended audience (bf:intendedAudience) for these photographs might be "general public". However this intended audience designation applies only to the photographs and not the resource at large.
  • A motion picture may have, as a part, a negative (or internegative) for which the extent (bf:extent) is to be described, for example "1 reel of 1 (37 ft.)". The extent applies only to the negative.

Questions:

  1. In Example 1, the abstract is represented as a Work. Should it be represented as such or should a class bf:Part be defined?
  2. In this example the type of part - abstract - is discernable only because it is included in the title, and it is most likely not machine discernible. Is this sufficient? If class bf:Part is defined (as suggested in question 1) should subclasses be defined for various part types?
  3. Although many parts can be represented as a separate resource (an abstract, table of contents, libretto), there are some parts that cannot, for example sung (or spoken) text, subtitles, and captions. Is it necessary to address these, and if so, how?
