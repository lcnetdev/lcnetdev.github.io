Changes for MODS Version 3.4 with draft user guidelines

The following changes for MODS Version 3.4 have been approved by the MODS Editorial Committee. Draft guidelines are included after each change, to be incorporated into the MODS User Guidelines after review by the MODS community. Text in red is for the MODS-EC to consider before this is released to the MODS community.

Functional changes

  • @naneTitleGroup on <name> and <titleInfo>
  • Bind a specific name to a specific title to enable creation of an author/title Uniform title. A new attribute, "nameTitleGroup" is added to both <titleInfo> and <name>. The data type of this attribute is string. To link a title and a name, MODS implementers will add the same value to @nameTitleGroup on the relevant <titleInfo> and <name> elements.

    Draft guidelines:
    nameTitleGroup - This attribute is used to link a top level <name> element and a top level <titleInfo> element for the purpose of encoding a name-title heading. The same value is assigned to this attribute used for the two elements. The recommended practice is to assign a value for each unique name within the pool of name-title headings in a record. This practice will acommodate cases where the same name is part of more than one name-title heading in a record.

    For example, if these two name-title headings are used in a record,

    Twain, Mark, 1835-1910. Adventures of Tom Sawyer.
    Twain, Mark, 1835-1910. Adventures of Huckleberry Finn.

    assign the nameTitleGroup value "1" for Mark Twain, and encode them this way

    <name type="personal" nameTitleGroup="1">
    <namePart>Twain, Mark</namePart>
    <namePart type="date">1835-1910</namePart>
    </name>
    <titleInfo type="uniform" nameTitleGroup="1">
    <title>Adventures of Tom Sawyer</title>
    </titleInfo>
    <titleInfo type="uniform" nameTitleGroup="1">
    <title>Adventures of Huckleberry Finn</title>
    </titleInfo>

    [Explanation of name-title encoding follows:]

    Encoding a name-title heading in MODS

    The MODS approach to encoding a name-title heading is to use separate top level <name> and <titleInfo> elements and link those two elements with the nameTitleGroup attribute (note that name-title headings as subjects are treated differently; see below). The value of that nameTitleGroup attribute is defined as a string. A value must be assigned for each unique name within the pool of name-title headings in a record. Such an assignment avoids complications that derive from the fact that a particular name may be part of more than one name-title heading in a record.

    Other features of MODS name-title heading encoding:

    • the title portion of a name-title heading is always a uniform title, so <titleInfo> always has the attribute type="uniform"
    • the title portion of a name-title heading is the distinctive part (the name portion of a name-title heading, on the other hand, may be used in more than one name-title heading in a record), so <titleInfo> is where authorityURI and valueURI attributes (or any other identifier values) for the name-title heading are placed

    Name-title headings that are used as subject headings are wrapped with the top level <subject> element; the nameTitleGroup attribute is not used in that case. For a name-title heading as subject, the <titleInfo> element should always have the type="uniform" attribute. However, an authority identifier for the name-title header should appear on the top level <subject> element, instead of its <titleInfo> child.

    Assigning usage="primary" for main/preferred entry and considerations for name-title headings

    In MODS, the usage="primary" attribute can be used to indicate a main/preferred entry. The nameTitleGroup attribute is used to link a top level <name> and a top level <titleInfo> to encode a name-title heading. Sometimes a name-title heading is the main/preferred entry. The title portion of a name-title heading is the more distinct part (whereas a particular name may sometimes be associated with more than one uniform title in a record), so the indication of main/preferred entry is assigned to the top level <titleInfo> element [see example #1].

    The usage="primary" attribute should only be assigned to a top level <name> element if it represents a name heading that is the main/preferred entry for the record. There may be rare cases when a name heading is the main/preferred entry, and that same name is part of one or more name-title headings (added/alternative) in the record. In that rare case, both usage="primary" and nameTitleGroup attributes will appear for the <name> [See example #2]. Apart from that case, if a top level <name> element has the nameTitleGroup attribute, it should not also have the usage="primary" attribute.

    Examples

    1) A name-title is main/preferred entry, and its same name component is part of another name-title heading in the record:

    Twain, Mark, 1835-1910. Adventures of Tom Sawyer. [main/preferred entry]
    Tom Sawyer; and, Huckleberry Finn [title of the resource]
    Twain, Mark, 1835-1910. Adventures of Huckleberry Finn. [added/alternative entry]

    Encoded in MODS this way:

    <titleInfo>
    <title>Tom Sawyer</title>
    <subTitle>and, Huckleberry Finn</subTitle>
    </titleInfo>
    <titleInfo type="uniform" usage="primary" nameTitleGroup="1">
    <title>Adventures of Tom Sawyer</title>
    </titleInfo>
    <titleInfo type="uniform" nameTitleGroup="1">
    <title>Adventures of Huckleberry Finn</title>
    </titleInfo>
    <name type="personal" nameTitleGroup="1">
    <namePart>Twain, Mark</namePart>
    <namePart type="date">1835-1910</namePart>
    </name>

    2) Name is main/preferred entry, but that same name is also part of name-title headings in the record:

        Crumb, George. [main/preferred entry]
        Chamber works of George Crumb and a public lecture by the composer [title of the resource]
        Crumb, George. Madrigals. Book I. [added/alternative entry]
        Crumb, George. Vox balaenae. [added/alternative entry]
        Crumb, George. Music: does it have a future? [added/alternative entry]

        Encoded in MODS this way:

          <titleInfo>
             <title>Chamber works of George Crumb and a public lecture by the composer</title>
          </titleInfo>
          <titleInfo type="uniform" nameTitleGroup="1">
             <title>Madrigals. Book I</title>
          </titleInfo>
          <titleInfo type="uniform" nameTitleGroup="1">
             <title>Vox balaenae</title>
          </titleInfo>
          <titleInfo type="uniform" nameTitleGroup="1">
             <title>Music: does it have a future?</title>
          </titleInfo>
          <name type="personal" usage="primary" nameTitleGroup="1">
             <namePart>Crumb, George</namePart>
          </name>

  • Add attribute "usage" with the value "primary" to the following MODS elements to indicate which or repeated elements are most important: titleInfo, name, typeOfResource, language, subject, classification, genre

    Draft guidelines:
    Usage - this attribute is used when there are repeated elements to indicate which should be used for primary display and/or indexing purposes

    Examples:

    <mods:genre usage="primary">Popular song</mods:genre>
    <mods:genre>Sheet music</mods:genre>

    <mods:name type="personal" usage="primary">
    <mods:namePart>Manoloff, Nick</mods:namePart>
    <mods:role>
    <mods:roleTerm type="code" authority="marcrelator">arr</mods:roleTerm>
    <mods:roleTerm type="text" authority="marcrelator">Arranger</mods:roleTerm>
    </mods:role>
    </mods:name>
    <mods:name type="personal">
    <mods:namePart>Field, Carl</mods:namePart>
    <mods:role>
    <mods:roleTerm type="code" authority="marcrelator">lyr</mods:roleTerm>
    <mods:roleTerm type="text" authority="marcrelator">Lyricist</mods:roleTerm>
    </mods:role>
    </mods:name>

  • Link translations and transliterations through a shared attribute. Each MODS top-level element will now have a new attribute, @altRepGroup.

    Draft guidelines:

    Addthe following to "Attributes Used Throughout the MODS Schema": linking attributes section:

    altRepGroup- This attribute is used to link alternate representations of the same element content, such as original script, transliterations, and translations. The attribute takes a string value, and the same value is applied to each of the element instances to be linked.

    Example:

    <name type="personal" altRepGroup="8">
    <namePart>Būrī, Muḥammad al-Tihāmī,</namePart>
    <namePart type="date">d. 1827</namePart>
    </name>
    <name type="personal" altRepGroup="8">
    <namePart>بوري، محمد التهامي،</namePart>
    <namePart type="date">d. 1827</namePart>
    </name>

    altRepGroup applies to MODS top-level elements with the exception of relatedItem. It also applies to those top-level elements when used within relatedItem.

    Care should be taken when mapping from MARC to MODS to insure that elements that are split from one field in MARC into two fields in MODS are linked correctly (e.g., MARC 245). Three conditions may occur in this case:  1) multiple pairs of linked elements (this is correct), 2) linked elements with identical content, because the alternative representation only occurs in a different portion of the MARC field, and 3) single MODS elements with an unmatched altRepGroup, if one of the linked MARC fields contained a subfield that the other MARC field did not (e.g. 245 $c).

  • Add lang, xml:lang, script and transliteration attributes to all textual MODS elements: This will introduce more flexibility for adding specific elements to a MODS record using multiple languages.

RDA-related changes

  • Add authorityURI and valueURI attributes to elements that have authority attribute
  • Add authorityURI and valueURI attributes to subject subelements
    Add the following to Attributes used throughout the schema section:
    Draft guidelines (for 2 above):
    • authorityURI – A URI uniquely identifying the vocabulary from which the controlled term has been selected, as assigned by the body responsible for the maintenance of the vocabulary. URIs identifying authorities may or may not be dereferenceable to human- or machine-readable information on the authority file, controlled vocabulary, or thesaurus.
    • valueURI – A URI uniquely identifying the term or controlled value, as assigned by the body responsible for the maintenance of the vocabulary. URIs identifying terms may or may not be dereferenceable to human- or machine-readable records for the term.
      Note for EC: need to address Jenn's change suggestions for subject subelements documented on wiki
  • Add value "family" to the attribute "type" under mods:name to indicate that the named entity is a family
    Draft guidelines:
    family - This attribute is used with the <name> element to express familial authorship. This type of authorship is common with archival materials. In addition, Resource Description and Access and Functional Requirements for Authority Data consider families as entities that can be creators of resources.

     Example:

     <name type="family">
    <namePart>Brooks family</namePart>
    <role><roleTerm type="text" authority="marcrelator">creator</roleTerm></role>
    </name>

  • Add values "single unit ", "multipart monograph", "serial", "integrating resource" as enumerated values for mods:issuance. The more general "monographic" and "continuing" will be kept if such distinctions are not needed.

    Draft guidelines:

    <issuance> may be used with the following values:

    • continuing - A resource that is issued over time with no predetermined conclusion. Continuing resources include serials and ongoing integrating resources, i.e. those that are continuously updated. Alternatively, more specific values distinguishing "serial" from "integrating resource" may be used (see below).
    • monographic - A resource that is either complete in one part or intended to be completed in a finite number of separate parts. Alternatively, more specific values distinguishing "single unit" from "multipart monograph" may be used (see below).
    • single unit - A resource that is issued either as a single physical unit (e.g., as a single-volume monograph) or, in the case of an intangible resource, as a single logical unit (e.g., as a PDF file mounted on the Web).
    • multipart monograph - A resource issued in two or more parts (either simultaneously or successively) that is complete or intended to be completed within a finite number of parts (e.g., a dictionary in two volumes or three audiocassettes issued as a set).
    • serial - A resource issued in successive parts, usually bearing numbering, that has no predetermined conclusion (e.g., a periodical, a monographic series, or a newspaper). Includes resources that exhibit characteristics of serials, such as successive issues, numbering, and frequency, but whose duration is limited (e.g., newsletters of events) and reproductions of serials
    • integrating resource - A resource that is added to or changed by means of updates that do not remain discrete and are integrated into the whole. An integrating resource may be tangible (e.g., a loose-leaf manual that is updated by means of replacement pages) or intangible (e.g., a Web site that is updated either continuously or on a cyclical basis).

      Collection, which designates an aggregation level, is included in the collection attribute in the <typeOfResource> element.

  • Define <scriptTerm> under <language> with attributes type (code, text), authority, authorityURI and valueURI. This will allow explicit and standardized indication (using iso15924) of the script used in a resource.

    Draft guidelines:
    Use <scriptTerm> to indicate the script in which the language is rendered. As with <languageTerm>, <scriptTerm> may be repeated to convey different representations of the that same script.

    Examples: 

    <abstract> ... work on ethics in 3 books ... followed by two short treatises ... the first of which is in Arabic ... and the second is in Arabic and Ottoman ....</abstract>
    <language objectPart="Primary work and first treatise">
    <languageTerm authority="iso639-2b" type="code">ara</languageTerm>
    <languageTerm lang="en" type="text">Arabic</languageTerm>
    <scriptTerm authority="iso15924" type="code">Arab</scriptTerm>
    </language>

    <language objectPart="Second treatise">
    <languageTerm authority="iso639-2b" type="code">ota</languageTerm>
    <languageTerm lang="en" type="text">Ottoman Turkish</languageTerm>
    <scriptTerm authority="iso15924" type="code">Arab</scriptTerm>
    </language>

    <language objectPart="Second treatise">
    <languageTerm authority="iso639-2b" type="code">ara</languageTerm>
    <languageTerm lang="en" type="text">Arabic</languageTerm>
    <scriptTerm authority="iso15924" type="code">Arab</scriptTerm>
    </language>

    <abstract>A Persian-Turkish literary dictionary</abstract>
    <language> <languageTerm authority="iso639-2b" type="code">per</languageTerm>
    <languageTerm authority="rfc3066" type="code">fa</languageTerm>
    <scriptTerm authority="iso15924" type="code">Arab</scriptTerm>
    <scriptTerm type="text">Arabic</scriptTerm>
    </language>

    <language>
    <languageTerm authority="iso639-2b" type="code">ota</languageTerm>
    <scriptTerm authority="iso15924" type="code">Arab</scriptTerm>

General/structural changes

  • Note to EC: is there any need to add anything to the Guidelines for any of these changes? I think these are just schema issues.
  • Enumerate the MODS version attribute to allow for specific indication of the particular MODS schema version used in the instance (currently the schema only includes the value "3.4" but will now also allow for "3.0", "3.1", "3.2", "3.3".)
  • Add <physicalDescription><extent> as a global element to the MODS namespace but not <part><extent>. It is not possible in version 3.4 to harmonize the definitions, since <extent> in these two places is very different, and changing either of them is too big a change for a minor release version. The short term-solution for MODS 3.4 is to only put physicalDescription/extent into the MODS namespace and not part/extent. In MODS 4.0 the definitions will be harmonized or one of these elements renamed so that both can be in the MODS namespace.
  • Reorganize the schema to externalize all elements and definitions and make the schema easier to read.
  • Define explicitly all elements in the MODS namespace (i.e. as "global" elements) so that they are addressable individually to other schemas and information systems. This requires the definitions of elements with the same name to have the same definition.

Miscellaneous changes

  • Add values "edtf" and "temper" to the attribute "encoding" under originInfo dates. This indicates use of the Extended Date Time Format or the TEMPER format.

    Draft guidelines:
    • edtf – This value is used for dates coded according to the "Extended Date Time Format" which can express special forms of dates that are not covered by other defined encodings such as w3cdtf and iso8601. Examples include: range, open ended: 2004/open. See http://www.loc.gov/standards/datetime/ for further details.
    • temper – This value is used for dates coded according to the “Temper date format” http://www.ietf.org/internet-drafts/draft-kunze-temper-01.txt. TEMPER (TEMPoral Enumerated Ranges) is a simple date and time syntax for representing points, lists, and ranges of timestamps. The syntax is designed to be trivial to parse, easy for humans to read, and friendly to basic lexical sorting algorithms. Examples include Non-Gregorian Calendar dates. A TEMPER point may be preceded by the three letters "BCE" for "Before Common Era" dates.  These three ASCII letters express (with no case sensitivity) "negative" dates, namely, dates that are chronologically less than the year 0000.  Examples include: BCE1212 Death of Rameses the Great.

      <dateOther> - Example

      <originInfo>

       

      <dateOther encoding="edtf">unknown/2009</dateOther>

      </originInfo>

       <subject> <temporal> - Example

      <subject>

       

      <temporal encoding="edtf">20050705/20050706</temporal>

      </subject>

  • Add attribute "supplied" with a single value "yes" to these elements: originInfo/edition, originInfo/place, originInfo/publisher, physicalDescription/extent, and titleInfo. This will mark data as having come from an outside source and that it is possibly questionable. (Possible changes to the "qualifier" attribute on dates may be forthcoming in a later version for consistency.)

    Draft guidelines:
    Supplied - This attribute is used to indicate that the value is from an external source from the resource (such as a cataloger)
    Example:
    <mods:originInfo>
    <mods:edition supplied="yes">Second</mods:edition>
    <mods:place supplied="yes">
    <mods:placeTerm type="text">Chicago</mods:placeTerm>
    </mods:place>
    <mods:publisher supplied="yes">Calumet Music</mods:publisher>
    <mods:copyrightDate encoding="w3cdtf" keyDate="yes">1935</mods:copyrightDate>
    </mods:originInfo>

  • Add values "references" and "review of" to relatedItem attribute "type". Mechanisms for extending the list of relatedItem types will be considered for MODS version 4.0.

    Draft guidelines:
    Add to type attribute values:
    references - Information concerning a resource cited or referred to in the resource.
    reviewOf - Information concerning a resource reviewed in the content of the resource. 

    Example:
    <mods:relatedItem type="reviewOf">
    <mods:titleInfo>The handbook of 20th century literature</mods:titleInfo>
    </mods:relatedItem>
